## 一、总体目标与范围

### 1.1 本期目标（第一阶段 MVP）

- 构建云端 FastAPI 单应用，提供任务/设备/项目管理 API
- 在各设备上部署 Python 极简 Bridge，实现 60 秒轮询 + Agent 执行
- 支持 `remote_execution` 类 Agent（claude_code / openclaw / hermes_agent / codex）自动执行
- 支持 `manual_only` 类 Agent（windsurf）企业微信 Webhook 提醒
- 提供基础 Web Dashboard（Next.js 15+ App Router + TailwindCSS + shadcn/ui）
- 实现项目停滞提醒 + 任务超时提醒

### 1.2 本期不做（第二阶段）

- 设备自主抢单 / 负载均衡
- WebSocket 实时推送
- Redis 队列
- 文件活动监控（watchdog）
- 多用户权限体系
- 邮件 / Slack 提醒渠道

### 阶段 2：核心 API 实现（第 2-3 天）

#### T2-1：设备管理 API

| 端点 | 说明 | 关键逻辑 |
|------|------|---------|
| `POST /api/v1/machines` | 注册设备 | `machine_id` 唯一校验；已存在则更新信息 |
| `GET /api/v1/machines` | 获取设备列表 | 支持 `status` / `agent_type` 过滤参数 |
| `GET /api/v1/machines/dashboard` | 获取仪表盘数据 | 返回所有设备及其运行中任务、今日完成数 |
| `GET /api/v1/machines/{id}` | 获取设备详情 | 返回设备 + 关联的未完成任务数量 |
| `PATCH /api/v1/machines/{id}` | 更新设备状态 | 支持更新 `is_enabled`（启用/禁用）和 `status` |
| `GET /api/v1/machines/{id}/poll` | 设备轮询获取任务 | 更新 `last_poll_at`；支持 `?project_id=` 按项目过滤；返回状态为 `pending` 的任务列表 |

**设备状态管理**：
- `status`：在线状态（online/offline），由心跳维持
- `is_enabled`：可用性（enabled/disabled），管理员可手动控制，禁用后不接收新任务

**轮询端点核心逻辑**：
```python
# 1. 更新 last_poll_at = NOW()
# 2. 更新 machine status = 'online'
# 3. 查询 tasks WHERE target_machine_id = machine.id AND status = 'pending'
#    （可选 WHERE project_id = :project_id 支持按项目过滤）
# 4. 将返回的任务批量更新为 status = 'dispatched'
# 5. 返回任务列表
```

**后台定时任务（APScheduler，每 2 分钟）**：
```python
# 将 last_poll_at 超过 120 秒的设备标记为 offline
```

#### T2-2：任务管理 API

| 端点 | 说明 | 关键逻辑 |
|------|------|---------|
| `POST /api/v1/tasks` | 创建任务 | 自动生成 `task_id`（`task-{uuid[:8]}`）；核心字段为 `instruction`（指令）；校验 `target_machine_id` 存在 |
| `GET /api/v1/tasks` | 获取任务列表 | 支持 `status` / `project_id` / `target_machine_id` 过滤；分页（`limit` + `offset`） |
| `GET /api/v1/tasks/{id}` | 获取任务详情 | 返回完整字段含 `result` JSONB |
| `PUT /api/v1/tasks/{id}` | 更新任务状态 | 仅允许更新 `status`；记录状态变更时间戳 |
| `POST /api/v1/tasks/{id}/result` | 提交执行结果 | Bridge 上报执行结果；更新 `result` / `completed_at` / `status` |
| `POST /api/v1/tasks/{id}/callback` | 设备回调上报结果 | 设备执行完成后通过 Hook 脚本调用；更新 `result` / `completed_at` / `status` |

**状态流转规则**：
```
pending → dispatched（轮询时服务端自动变更）
dispatched → running（Bridge 开始执行时上报）
running → completed / failed（Bridge 执行结束时上报）
pending / dispatched / running → cancelled（用户主动取消）
pending_manual → completed（用户手动标记完成）
```

**任务模型（精简版）**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `instruction` | string | 执行指令（核心字段） |
| `project_id` | uuid | 关联的项目 |
| `target_machine_id` | uuid | 指定的执行设备 |
| `status` | enum | pending / dispatched / running / completed / failed / cancelled |
| `result` | json | 执行结果：{exit_code, stdout, stderr, error_type} |
| `error_message` | string | 错误信息 |

已移除：`title`、`description`、`priority`、`deadline`、`timeout_seconds`

#### T2-3：项目管理 API

| 端点 | 说明 | 关键逻辑 |
|------|------|---------|
| `POST /api/v1/projects` | 创建项目 | 自动生成 `project_id` |
| `GET /api/v1/projects` | 获取项目列表 | 返回每个项目的 `last_activity_at`、停滞小时数、是否超过 `idle_threshold_hours` |
| `PUT /api/v1/projects/{id}` | 更新项目 | 支持归档（`is_archived = true`）、修改阈值 |

**项目自动创建**：终端（Bridge）在注册时可通过 `project_id` 指定关联项目，项目不存在时自动创建。

#### T3-3：api_client.py — API 客户端

实现以下方法：
```python
async def register_machine()          # 启动时注册本机
async def poll_tasks(project_id=None) -> list[Task]  # 拉取待执行任务，支持按项目过滤
async def update_task_status(task_id, status)
async def submit_result(task_id, result: dict)
async def send_reminder(task_id)      # 触发服务端发送微信提醒
```

- 所有请求携带 `X-API-Key` Header
- 网络超时设置：connect=5s，read=30s
- 失败时 `loguru` 记录错误，不抛出异常（确保轮询循环不中断）
- 重试策略：网络错误最多重试 3 次，间隔 2s

#### T3-4：executor.py — Agent 执行器

**基类**：
```python
class AgentExecutor:
    async def execute(self, instruction: str, workdir: str) -> dict
    # 返回: {"exit_code": int, "stdout": str, "stderr": str, "error_type": str | None}
```

**具体实现**（根据 T0-1 预研结果填写实际命令）：

| 类 | Agent | 命令模板 |
|----|-------|---------|
| `ClaudeCodeExecutor` | claude_code | `claude --print "{instruction}"` |
| `OpenClawExecutor` | openclaw | 待 T0-1 确认 |
| `HermesAgentExecutor` | hermes_agent | 待 T0-1 确认 |
| `CodexExecutor` | codex | 待 T0-1 确认 |

- 工作目录：`/tmp/dus_task_{task_id}/`（执行前创建，执行后保留日志）
- 超时：从 `task.timeout_seconds` 读取（默认 3600s）
- 超时后 `kill` 子进程，返回 `error_type = "timeout"`

#### T3-5：main.py — 主轮询循环

```python
async def main():
    await api.register_machine()          # 启动注册
    while True:
        tasks = await api.poll_tasks()
        for task in tasks:
            asyncio.create_task(handle_task(task))  # 并发执行多任务
        await asyncio.sleep(config.poll_interval)   # 默认 60s

async def handle_task(task):
    if task.agent_capability == "remote_execution":
        await execute_remote(task)
    elif task.agent_capability == "manual_only":
        await send_manual_reminder(task)
```

- 并发上限：`asyncio.Semaphore(3)`（最多同时执行 3 个任务）
- `Ctrl+C` 优雅退出，等待正在执行的任务完成

### 阶段 4：提醒系统（第 5-6 天）

#### T4-1：企业微信 Webhook 发送模块（云端）

创建 `cloud/notifier.py`：
```python
async def send_wechat_markdown(title: str, content: str) -> bool
```

- 使用 `httpx.AsyncClient` POST 到 Webhook URL
- 发送失败时记录日志，不影响主流程
- 发送结果（成功/失败）写回 `task.result["reminder_sent"]`

#### T4-2：Windsurf 任务提醒

触发时机：Bridge 轮询到 `manual_only` 设备的任务时，调用 `POST /api/v1/tasks/{id}/remind`

云端处理：
1. 更新任务状态为 `pending_manual`
2. 调用 `notifier.send_wechat_markdown()`，使用模板：
   ```
   🤖 Windsurf 任务提醒
   任务：{task_title}
   设备：{machine_name}
   项目：{project_name}
   目录：{project_root_path}
   指令：{instruction}
   💡 完成后请手动标记任务完成
   🔗 [查看任务]({task_link})
   ```

#### T4-3：APScheduler 定时提醒（云端）

在 `cloud/scheduler.py` 中注册两个定时任务：

| 任务 | 频率 | 逻辑 |
|------|------|------|
| 项目停滞检查 | 每 1 小时 | 查询 `last_activity_at < NOW() - idle_threshold_hours`，未归档项目发送停滞提醒 |
| 任务超时检查 | 每 5 分钟 | 查询 `status = 'running' AND started_at + timeout_seconds < NOW()`，发送超时提醒并标记 `failed` |

提醒去重：在 `task.result` 中记录最后提醒时间，间隔不足 `reminder_interval_hours` 不重复发送。

- 轮询间隔：**3 秒**（任务运行中时，TanStack Query `refetchInterval` 动态调整）

#### T5-6：项目状态页 `/projects`

**功能**：
- 展示所有项目卡片
- 每个卡片显示：项目名、项目路径、最后活动时间、停滞时长、停滞状态（正常/警告/超限）
- 停滞状态颜色规则：
  - 未超 `idle_threshold_hours`：绿色
  - 超过阈值但在 2 倍以内：黄色警告
  - 超过 2 倍阈值：红色超限
- 轮询间隔：**30 秒**

#### T5-7：状态徽章组件 `StatusBadge`（shadcn Badge 变体）
